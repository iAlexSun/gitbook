autorelease原理

简要说一下 autoreleasePool 的数据结构

autoreleasepool 以一个队列数组的形式实现,主要通过下列三个函数完成.

  • objc_autoreleasepoolPush

  • objc_autoreleasepoolPop

  • objc_autorelease

看函数名就可以知道，对 autorelease 分别执行 push，和 pop 操作。销毁对象时执行release操作。

void *objc_autoreleasePoolPush(void) {
    return AutoreleasePoolPage::push();
}

void objc_autoreleasePoolPop(void *ctxt) {
    AutoreleasePoolPage::pop(ctxt);
}

autoreleasePool是由若干个AutoreleasePoolPage组成双向链表链接而成, AutoreleasePoolPage的结构如下

class AutoreleasePoolPage  {
   magic_t const magic; // 用来校验AutoreleasePoolPage当前结构的完整性
   id *next;            
   pthread_t const thread;  //当前在哪个线程
   AutoreleasePoolPage * const parent;   //父节点指针
   AutoreleasePoolPage *child;    //子节点指针
   uint32_t const depth;
   uint32_t hiwat;
}

AutoreleasePoolPage在栈中如何存储的，假设 0x100816000 ~ 0x100817000是用来存储AutoreleasePoolPage的地址，那么其中有 56 bit 用于存储 AutoreleasePoolPage 的成员变量，也就是0x100816000~ 0x100816038，剩下的地址0x100816038 ~ 0x100817000用来存储添加到AutoreleasePoolPage中的释放对象

next指针是用来指向下一个空的内存地址，如果添加进来一个需要释放的对象，那么next指针就需要向后移动一个位置到下一个内存为空的地址

POOL_SENTINEL(哨兵对象)

#define POOL_SENTINEL nil

这里定义了POOL_SENTINEL为一个nil，其实就是一个哨兵对象，作用就是用来标记如果objc_autoreleasePoolPop操作的时候，向每个在释放池中对象进行发送release消息，当弹出到第一个POOL_SENTINEL对象时就停止，在每个自动释放池初始化调用 objc_autoreleasePoolPush 的时候都会将POOL_SENTINEL对象放在栈顶。

int main(int argc, const char * argv[]) {
  {
      void * atautoreleasepoolobj = objc_autoreleasePoolPush();
      // do whatever you want
      objc_autoreleasePoolPop(atautoreleasepoolobj);
  }
    return 0;
}

为什么已经有了 ARC ,但还是需要 @AutoreleasePool 的存在

可能瞬时产生内存暴增的情况，比如下载图片时生成大量UIImage对象,不能及时释放就会产生内存暴增，如果使用AutoreleasePool在可以在执行完成会就会当前Runloop结束后就会释放。

for(int i=0;i<999999;i++){
    //这里需要添加,如果不添加@AutoreleasePool可能会出现内存一直增加的情况出现,
    //这里添加@ AutoreleasePool将内存及时的释放，也可以使用enumerateObjectsUsingBlock代替，
    因为enumerateObjectsUsingBlock内部已经实现了@AutoreleasePool操作，所以就不会出现内存暴增的问题
    @AutoreleasePool {
        //处理事务
        UIImage *image = [UIImage new];
    NSLog(@"%d",i);]
    }
}

经过测试如果不添加@AutoreleasePool可能从8M增涨到40M，如果在循环中操作更多操作如NSData->UIImage，image解码等操作可能会出现内存暴增几倍甚至几十倍，可能出现程序崩溃的情况。

SDWebImage中使用了@AutoreleasePool

13.函数返回一个对象时，会对对象 autorelease 么？为什么？

以alloc init方式进行初始化不会在方法返回的时候添加autorelease,在作用域结束后就会release

{
 id __strong obj = [[NSMutableArray alloc]init];
}

结果转化后

//编译器的模拟代码
id obj = objc_msgSend(NSObject,@selector(alloc));
objc_msgSend(obj,@selector(init));

// 出作用域的时候调用
objc_release(obj);

对象是通过除alloc、new、copy、multyCopy外方法产生的情况

ARC情况下
{
    id __strong obj = [NSMutableArray array];
}

MRC情况下
{
    id obj = [NSMutableArray array];
    [obj retain];
    [obj release];
}

结果转换为

//编译器的模拟代码
id obj = objc_msgSend(NSMutableArray,@selector(array));
objc_retainAutoreleasedReturnValue(obj);
objc_release(obj);

目前经过编译器的优化会变成objc_retainAutoreleasedReturnValue，，所以在外部需要手动retain计数加 1 持有当前对象。 经过编译器优化后使用objc_retainAutoreleasedReturnValue函数。

因为在MRC时类方法生成但是不持有，所以在mrc的情况下调用类方法需要手动retain一下， 因为在类方法的内部实现会调用autorelease进行然后返回当前对象。

ARC代码
+ (id)array {
    return [[NSMutableArray alloc] init];
}

MRC代码
+ (id) array {
    //自己持有对象
    id obj = [[NSMutableArray alloc]init];
    [obj autorelease];
    //取得的对象存在，但自己不持有对象
    return obj;
}

结果转换为

//编译器模拟代码
+ (id)array {
    id obj = objc_msgSend(NSMutableArray,@selector(alloc));
    objc_msgSend(obj,@selector(init));

    // 代替我们调用了autorelease方法
    return objc_autoreleaseReturnValue(obj);
}

我们可以看见调用了objc_autoreleaseReturnValue函数且这个函数会返回注册到自动释放池的对象，但是，这个函数有个特点，它会查看调用方的命令执行列表，如果发现接下来会调用objc_retainAutoreleasedReturnValue则不会将返回的对象注册到autoreleasePool中而仅仅返回一个对象。达到了一种最优效果。如下图

有了这个基础，ARC 可以使用一些优化技术。在调用 objc_autoreleaseReturnValue()时，会在栈上查询 return address以确定 return value 是否会被直接传给 objc_retainAutoreleasedReturnValue()。 如果没传，说明返回值不能直接从提供方发送给接收方，这时就会调用 autorelease。反之，如果返回值能顺利的从提供方传送给接收方，那么就会直接跳过 autorelease 过程，并且修改return address 以跳过 objc_retainAutoreleasedReturnValue()过程，这样就跳过了整个autorelease和 retain的过程。

核心思想：当返回值被返回之后，紧接着就需要被retain的时候，没有必要进行autorelease + retain，直接什么都不要做就好了。

@autoreleasrPool 的释放时机？

App启动后，苹果在主线程RunLoop里注册了两个 Observer，其回调_wrapRunLoopWithAutoreleasePoolHandler()。

1.<CFRunLoopObserver>{activities = 0x1, callout = _wrapRunLoopWithAutoreleasePoolHandler} 
2.<CFRunLoopObserver>{activities = 0xa0, callout = _wrapRunLoopWithAutoreleasePoolHandler}

  • 第一个 Observer 监视的事件是 activities = 0x1 kCFRunLoopEntry(即将进入Loop)，其回调内会调用 _objc_autoreleasePoolPush() 创建自动释放池。其 order 是-2147483647，优先级最高，保证创建释放池发生在其他所有回调之前。

  • 第二个 Observer 监视了activities = 0xa0两个事件：

    • kCFRunLoopBeforeWaiting(准备进入休眠) 时调用_objc_autoreleasePoolPop()和 _objc_autoreleasePoolPush() 释放旧的池并创建新池；

    • kCFRunLoopExit(即将退出Loop) 时调用 _objc_autoreleasePoolPop() 来释放自动释放池，这个 Observer 的 order 是 2147483647，优先级最低，保证其释放池子发生在其他所有回调之后。

在主线程执行的代码，通常是写在诸如事件回调、Timer回调内的。这些回调会被 RunLoop 创建好的 AutoreleasePool 环绕着，所以不会出现内存泄漏，开发者也不必显示创建 Pool 了。

访问 __weak 修饰的变量，是否已经被注册在了 @autoreleasePool 中？为什么？

添加在autoreleasePool中了,如果没有添加到autoreleasePool中就会被立即释放了,为了延长声明周期所以添加在了autoreleasePool中.

在objc_loadWeak可以找到

objc_loadWeak(id *location) {
    if (!*location) return nil;
    return objc_autorelease(objc_loadWeakRetained(location));
}
