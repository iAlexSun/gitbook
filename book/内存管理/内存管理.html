内存管理

iOS 内存管理的理解 (三种方案的结合)

  • 1.TaggedPointer（针对类似于 NSNumber 的小对象类型）

    这里会涉及到为什么要引入TaggedPointer这个概念，这里会涉及到如何内存布局，一个对象或属性在占用内存的大小来说，

    由于每个对象在64位系统下每个对象会占用16 字节，但是TaggedPointer只会占用 8字节，为了避免一些不必要的浪费，比如NSNumber类型的数字可能不需要，TaggedPointer 与 对象根本的区别在于，每个对象是一个isa指针类型，如果访问或者修改就要去寻找内存中的内存地址，但是TaggedPointer本质是直接访问内存地址进行修改及存储，所以占用内存会小于对象，对象的内存对齐是以16的倍数，在介绍类结构中会介绍相关部分。

  • 2.NONPOINTER_ISA（64位系统下）

    • 第一位的 0 或 1 代表是纯地址型 isa 指针，还是 NONPOINTER_ISA 指针。
    • 第二位，代表是否有关联对象
    • 第三位代表是否有 C++ 代码。
    • 接下来33位代表指向的内存地址
    • 接下来有 弱引用 的标记
    • 接下来有是否 delloc 的标记....等等
union isa_t {
    isa_t() { }
    isa_t(uintptr_t value) : bits(value) { }

    Class cls;
    uintptr_t bits;
#if defined(ISA_BITFIELD)
    struct {
        uintptr_t nonpointer        : 1; 
        uintptr_t has_assoc         : 1;                                       
        uintptr_t has_cxx_dtor      : 1;                                       
        uintptr_t shiftcls          : 33; /*MACH_VM_MAX_ADDRESS 0x1000000000*/ 
        uintptr_t magic             : 6;                                       
        uintptr_t weakly_referenced : 1;                                       
        uintptr_t deallocating      : 1;                                       
        uintptr_t has_sidetable_rc  : 1;                                       
        uintptr_t extra_rc          : 19
    };
#endif
};
  • 3.散列表（引用计数表、weak表）
    • SideTables 表在 非嵌入式的64位系统中，有 64张 SideTable 表
    • 每一张 SideTable 主要是由三部分组成。自旋锁、引用计数表、弱引用表。
    • 全局的 引用计数 之所以不存在同一张表中，是为了避免资源竞争，解决效率的问题。
    • 引用计数表 中引入了 分离锁 的概念，将一张表分拆成多个部分，对他们分别加锁，可以实现并发操作，提升执行效率

如果在64位系统下且为 nonpointer 类型的 isa 指针则引用计数首先会在 extra_rc 中存储，在64位系统下可以存储19位，在x86结构下只能存储8位。如果引用计数超过了19位之后 has_sidetable_rc 则会变为1，意味着需要新的引用计数也就是散列表中的 RefcountMap refcnts 引用计数表来存储引用计数。

使用自动引用计（ARC）数应该遵循的原则

  • 不能使用 retain、release、retainCount、autorelease。
  • 不可以使用 NSAllocateObject、NSDeallocateObject。
  • 必须遵守内存管理方法的命名规则。
  • 不需要显示的调用 dealloc。
  • 使用 @autoreleasePool 来代替 NSAutoreleasePool。
  • 不可以使用区域 NSZone。
  • 对象性变量不可以作为 C 语言的结构体成员。
  • 显示转换 id 和 void*。

说一下对 strong, copy, assign, weak,_ unsafe_unretained 关键字的理解。

  • strong

    指向对象且强引用当前对象，引用计数加1。改对象如果引用计数不为0不会自动释放，可以使用nil来进行释放。

  • weak

    指向对象但是弱引用，引用计数不加1，不会保持当前对象存活。如果当前对象释放后，则弱引用会被runtime置为nil。

  • copy

    一般用在有可变类型，会用在不可改变的类型如NSString，NSArray等类型。不过在赋值时进行 copy 操作而不是 retain 操作。通常在需要保留某个不可变对象（NSString最常见），并且防止它被意外改变时使用。

    //copy环境下
- (void)setName:(NSString *)name{
    if (_name != name) {
        [_name release];
        _name = [name copy];
    }
}

  • assign

    用来修饰基础数据类型，如CGFloat，NSInterger等，引用计数不加1，也不持有对象。也可以修饰对象，但是会出现内存问题。

  • _ unsafe_unretained

    与weak功能类似，存在于iOS4系统中，也是指向对象弱引用，引用计数也不加1。但是与weak的区别在于，如果弱引用的对象释放，unsafe_unretained指针可能会变成悬垂指针，产生内存问题。但是在效率上会比weak好很多，在YYModel中有使用_ unsafe_unretained

__weak 和 _unsafe_unretain 的区别

{
     id obj = [NSObject alloc]init]; //对象A
     _weak id obj1 = obj;
}
//作用域结束后对象A废弃，释放内存,
// obj强引用失效，释放内存
// ob1j弱引用，释放内存，且置为nil

__weak 不会持有对象，在指向的对象A释放后，用 __weak 修饰的对象obj也会释放且置为nil

{
    id obj = [NSObject alloc]init]; //对象B
    _Unsafe_Unretain id obj1 = obj;
}
NSLog(@"%@",obj1);
//作用域结束后对象A释放内存,obj 强引用失效，obj释放内存
// obj1 变量表示的对象obj已经释放，所以可能出现悬垂指针问题，BAD_ACCESS崩溃问题

_unsafe_unretain 也不会持有对象，在指向的对象B释放后，用_unsafe_unretain修饰的对象。weak 修饰的指针变量，在指向的内存地址销毁后，会在 Runtime 的机制下，自动置为 nil。_unsafe_unretain不会置为 nil，容易出现 悬垂指针，发生崩溃。但是 _unsafe_unretain 比 __weak 效率高。

ARC 做了哪些工作

编译时

​ 自动在合适的时机插入retain，release，autorelease

运行时

  • 使用weak修饰的变量引用计数为0时，利用runtime进行weak属性置为nil。

  • 为了保证向后兼容性，ARC 在运行时检测到类函数中的 autorelease 后紧跟其后 retain，此时不直接调用对象的 autorelease 方法，而是改为调用 objc_autoreleaseReturnValue。 objc_autoreleaseReturnValue 会检视当前方法返回之后即将要执行的那段代码，若那段代码要在返回对象上执行 retain 操作，则设置全局数据结构中的一个标志位，而不执行 autorelease 操作，与之相似，如果方法返回了一个自动释放的对象，而调用方法的代码要保留此对象，那么此时不直接执行 retain ，而是改为执行 objc_retainAoutoreleasedReturnValue函数。此函数要检测刚才提到的标志位，若已经置位，则不执行 retain 操作，设置并检测标志位，要比调用 autorelease 和retain更快。

什么是 悬垂指针？什么是野指针?

  • 悬垂指针 是指针指向的内存已经释放，但是该指针依旧存在，但是可能会出现内存错误。

  • 野指针 没有进行初始化的指针
