说一下 Method Swizzling? 说一下在实际开发中你在什么场景下使用过?

可以交换两个方法的实现，举个例子

+ (void)load {
    static dispatch_once_t onceToken;
    dispatch_once(&onceToken, ^{
        Class class = [self class];
        SEL originalSelector = @selector(run);
        SEL swizzledSelector = @selector(xxx_run);

        Method originalMethod = class_getInstanceMethod(class, originalSelector);
        Method swizzledMethod = class_getInstanceMethod(class, swizzledSelector);
        BOOL didAddMethod = class_addMethod(class,
                                            originalSelector,
                                            method_getImplementation(swizzledMethod),
                                            method_getTypeEncoding(swizzledMethod));
        if (didAddMethod) {
            class_replaceMethod(class,
                                swizzledSelector,
                                method_getImplementation(originalMethod),
                                method_getTypeEncoding(originalMethod));
        } else {
            method_exchangeImplementations(originalMethod, swizzledMethod);
        }
    });
}

- (void)xxx_run {
    NSLog(@"hook 跑起来");
    [self xxx_run];
}

一般情况下在+load方法中实现方法交换，使用单例的方式进行来保证方法只置换一次，大概的逻辑就是通过SEL去获取class_getInstanceMethod实现方法，一般情况下会先判断是否有这个方法存在，如果存在的话就置换这个方法，这Category中进行置换的方法实现，这里面会有一个需要注意的地方，就是一般交换后会再去执行原类的方法实现。

- (void)xxx_run {
    //先执行hook方法，再去执行原类的方法
    NSLog(@"hook 跑起来");
    [self xxx_run];

   //先去执行原类的方法, 再执行hook方法，
   [self xxx_run];
   NSLog(@"hook 跑起来");

   //替换原类方法的实现
   NSLog(@"hook 跑起来");

}

上面的方法看起来会出现死循环，但是其实不会出现死循环因为实现方法已经被交换，所以现在调用自己其实在调用的是hook的类的方法，这里还有hook的时机像上面的代码会有3种情况，先执行hook代码，先执行原类代码，替换原来代码。

实际应用

那么如果hook的是父类的代码会出现什么情况，因为在日常使用中需要hook UIViewController的viewWillAppear时机，如果hook 自定义的viewController与基类的会有不同的情况。如果hook父类且子类复写该方法会出现什么样的情况？

//父类
- (void)run {
     NSLog(@"跑起来");
}

//子类
- (void)run {
    [super run];
    NSLog(@"person3 跑起来");
}

2020-05-06 19:49:28.072571+0800 runtime[83806:6660927] 跑起来
2020-05-06 19:49:28.072691+0800 runtime[83806:6660927] person3 跑起来

2020-05-06 19:49:28.072453+0800 runtime[83806:6660927] 跑起来
2020-05-06 19:49:28.072571+0800 runtime[83806:6660927] hook 跑起来
2020-05-06 19:49:28.072691+0800 runtime[83806:6660927] person3 跑起来

如果hook父类的方法且子类集成父类方法那么hook的方法不会对子类有什么影响，父类的方法会先执行或者hook的方法先执行。父类方法->hook方法->子类方法，hook方法->父类方法->子类方法，hook方法->子类方法。

如果hook的是子类的方法那么就是父类方法->hook方法->子类方法，父类方法->子类方法->hook方法，父类方法->hook方法，三种情况。

所以在日常埋点的过程中需要hook UIViewController的`viewWillAppear方法，hook方法永远先执行于子类方法。

消息转发过程如上图，主要过程有如下三个

//查找当前类是否实现方法，如果没找到可以在这里进行动态添加一个其他方法进行执行，返回YES即可
+ (BOOL)resolveInstanceMethod:(SEL)sel {
    if ([NSStringFromSelector(sel) isEqualToString:@"say"]) {
         // 我们没有给People类声明sing方法，我们这里动态添加方法
        class_addMethod(self, sel, (IMP)otherSing, "v@:");
        return YES;
    }
    return [super resolveInstanceMethod:sel];
}

//实现方法
void otherSing(id self, SEL cmd) {
    NSLog(@"%@ 唱歌啦！",((Person *)self).name);
}

//如果返回NO那么就进行结束第一步的查找进入第二步
+ (BOOL)resolveInstanceMethod:(SEL)sel {
    return NO;
}

//这个主要是进行使用其他对象进行转发，拥有相同的aSelector即可实现转发对象调用其实现的同名方法则不会崩溃
- (id)forwardingTargetForSelector:(SEL)aSelector {
     forwardTest *forward = [forwardTest new];
    if ([NSStringFromSelector(aSelector) isEqualToString:@"say"]) {
        return forward;
    }
    return nil;
}
//如果没有转发对象那么，进入第三步
- (id)forwardingTargetForSelector:(SEL)aSelector {
        retutrn nil;
}

//第三步主要进行了方法签名进行第四步操作
- (NSMethodSignature *)methodSignatureForSelector:(SEL)aSelector {
      //这里如果不进行方法签名就会返回没有实现方法的签名，没有实现的方法签名为nil则会崩溃
    if ([NSStringFromSelector(aSelector) isEqualToString:@"say"]) {
        // 这里签名主要是方法的Type Encoding
        return [NSMethodSignature signatureWithObjCTypes:"v@:"];
    }
    return [super methodSignatureForSelector:aSelector];
}

// 第四步：这步我们修改调用方法
- (void)forwardInvocation:(NSInvocation *)anInvocation {
    [anInvocation setSelector:@selector(dance)];
    // 这还要指定是哪个对象的方法
    [anInvocation invokeWithTarget:self];
}
- (void)dance {
    NSLog(@"跳舞！！！come on！");
}

- (void)doesNotRecognizeSelector:(SEL)aSelector {
    NSLog(@"消息无法处理：%@", NSStringFromSelector(aSelector));
}
  • 第一步：通过resolveInstanceMethod：方法决定是否动态添加方法。如果返回Yes则通过class_addMethod动态添加方法，消息得到处理，结束；如果返回No，则进入下一步；
  • 第二步：这步会进入forwardingTargetForSelector:方法，用于指定备选对象响应这个selector，不能指定为self。如果返回某个对象则会调用对象的方法，结束。如果返回nil，则进入第三部；
  • 第三部：这步我们要通过methodSignatureForSelector:方法签名，如果返回nil，则消息无法处理。如果返回methodSignature，则进入下一步；
  • 第四部：这步调用forwardInvocation：方法，我们可以通过anInvocation对象做很多处理，比如修改实现方法，修改响应对象等，如果方法调用成功，则结束。如果失败，则进入doesNotRecognizeSelector方法，若我们没有实现这个方法，那么就会crash。

举一个普通的方法调用的例子如何转换成objc_msgSend

OBJC_EXPORT id objc_msgSend(id self, SEL op, ...)

forwardTest *forward = [forwardTest new];

//目前我们调用方法
[forward say];

//转换成 objc_msgSend 调用的形式
objc_msgSend(forward, @selector(say));

//或者转换成强制转换的形式
((void(*)(id, SEL)) objc_msgSend)((id)forward, @selector(say));

objc_msgSend没有开源所以只能靠推测内部是如何实现的。通过receiver forward去在 objc_msgSend 实现内部去找到当前的类，然后根据当前的selector去方法列表中去寻找是否有这个方法的实现。具体查找的方法在runtime源码中lookUpImpOrForward会有执行。暴露给最外面的方法是_class_lookupMethodAndLoadCache3,调用栈应该是这样的

0 lookUpImpOrForward
1 _class_lookupMethodAndLoadCache3
2 objc_msgSend
3 main
4 start

/***********************************************************************
* _class_lookupMethodAndLoadCache.
* Method lookup for dispatchers ONLY. OTHER CODE SHOULD USE lookUpImp().
* This lookup avoids optimistic cache scan because the dispatcher 
* already tried that.
**********************************************************************/
IMP _class_lookupMethodAndLoadCache3(id obj, SEL sel, Class cls) {
    return lookUpImpOrForward(cls, sel, obj, 
                              YES/*initialize*/, NO/*cache*/, YES/*resolver*/);
}

lookUpImpOrForward源码

IMP lookUpImpOrForward(Class cls, SEL sel, id inst, bool initialize, bool cache, bool resolver {
    IMP imp = nil;
    bool triedResolver = NO;

    runtimeLock.assertUnlocked();

    // Optimistic cache lookup
    if (cache) {
        imp = cache_getImp(cls, sel);
        if (imp) return imp;
    }
    runtimeLock.lock();
    checkIsKnownClass(cls);

    if (!cls->isRealized()) {
        realizeClass(cls);
    }

    if (initialize  &&  !cls->isInitialized()) {
        runtimeLock.unlock();
        _class_initialize (_class_getNonMetaClass(cls, inst));
        runtimeLock.lock();
    }

 retry:    
    runtimeLock.assertLocked();
    imp = cache_getImp(cls, sel);
    if (imp) goto done;
    {
        Method meth = getMethodNoSuper_nolock(cls, sel);
        if (meth) {
            log_and_fill_cache(cls, meth->imp, sel, inst, cls);
            imp = meth->imp;
            goto done;
        }
    }

  {
        unsigned attempts = unreasonableClassCount();
        for (Class curClass = cls->superclass;curClass != nil;curClass = curClass->superclass){
            // Halt if there is a cycle in the superclass chain.
            if (--attempts == 0) {
                _objc_fatal("Memory corruption in class list.");
            }

            // Superclass cache.
            imp = cache_getImp(curClass, sel);
            if (imp) {
                if (imp != (IMP)_objc_msgForward_impcache) {
                    // Found the method in a superclass. Cache it in this class.
                    log_and_fill_cache(cls, imp, sel, inst, curClass);
                    goto done;
                }
                else {
                    // Found a forward:: entry in a superclass.
                    // Stop searching, but don't cache yet; call method 
                    // resolver for this class first.
                    break;
                }
            }

            // Superclass method list.
            Method meth = getMethodNoSuper_nolock(curClass, sel);
            if (meth) {
                log_and_fill_cache(cls, meth->imp, sel, inst, curClass);
                imp = meth->imp;
                goto done;
            }
        }
    }

    // No implementation found. Try method resolver once.

    if (resolver  &&  !triedResolver) {
        runtimeLock.unlock();
        _class_resolveMethod(cls, sel, inst);
        runtimeLock.lock();
        // Don't cache the result; we don't hold the lock so it may have 
        // changed already. Re-do the search from scratch instead.
        triedResolver = YES;
        goto retry;
    }

    // No implementation found, and method resolver didn't help. 
    // Use forwarding.

    imp = (IMP)_objc_msgForward_impcache;
    cache_fill(cls, sel, imp, inst);

 done:
    runtimeLock.unlock();
    return imp;
}

下面来逐个分析每种情况的代码

无锁的缓存查找

IMP imp = nil;
bool triedResolver = NO;
runtimeLock.assertUnlocked();
if (cache) {
    imp = cache_getImp(cls, sel);
    if (imp) return imp;
}

这里去查找是否有缓存，去 cache_getImp 去找到当前方法的实现。但是进入的入口是_class_lookupMethodAndLoadCache3，传入的 cache == NO，所以这个 if 判断应该会被跳过不会执行这段代码。

类的实现和初始化

runtimeLock.lock();
checkIsKnownClass(cls);

if (!cls->isRealized()) {
    realizeClass(cls);
}

if (initialize  &&  !cls->isInitialized()) {
    runtimeLock.unlock();
    _class_initialize (_class_getNonMetaClass(cls, inst));
    runtimeLock.lock();
}

通过调用realizeClass方法，分配可读写class_rw_t的空间，主要进行了类的初始化操作，这里可以之后详细分析。

查找当前类实现

imp = cache_getImp(cls, sel);
if (imp) goto done;

主要是调用cache_getImp查找这个类的缓存中是否有这个方法的实现，与开始的去缓存中查询方法是相同的。如果查询到了方法直接结束执行done。

Method meth = getMethodNoSuper_nolock(cls, sel);
if (meth) {
    log_and_fill_cache(cls, meth->imp, sel, inst, cls);
    imp = meth->imp;
    goto done;
}

如果没有查找到方法缓存，因为我们第一次执行执行类中某个方法就会进入这段逻辑，那么会从类的方法列表中进行查询是否有这个方法，如果找到这个方法那么就缓存起来。

getMethodNoSuper_nolock 方法的实现

static method_t *getMethodNoSuper_nolock(Class cls, SEL sel) {
    for (auto mlists = cls->data()->methods.beginLists(),
              end = cls->data()->methods.endLists();
         mlists != end;
         ++mlists)
    {
        method_t *m = search_method_list(*mlists, sel);
        if (m) return m;
    }
    return nil;
}

父类查找方法

for (Class curClass = cls->superclass;curClass != nil;curClass = curClass->superclass){

基本与上面查找的方式相同，只是一直向上查找父类->根类中有没有方法的实现，如果在父类的缓存中找到_objc_msgForward_impcache会交给当前类处理。

没有找到方法实现

// No implementation found. Try method resolver once.
if (resolver  &&  !triedResolver) {
    runtimeLock.unlock();
    _class_resolveMethod(cls, sel, inst);
    runtimeLock.lock();
      //这里 triedResolver 会标记为YES,不会有下一次查找了
    triedResolver = YES;
    goto retry;
}

如果上面的方法都没有找到这个方法实现，那么就会调用_class_resolveMethod来解析没有实现的方法之后再去查找一次过程。

void _class_resolveMethod(Class cls, SEL sel, id inst) {
    if (! cls->isMetaClass()) {
        // try [cls resolveInstanceMethod:sel]
        _class_resolveInstanceMethod(cls, sel, inst);
    } else {
        // try [nonMetaClass resolveClassMethod:sel]
        // and [cls resolveInstanceMethod:sel]
        _class_resolveClassMethod(cls, sel, inst);
        if (!lookUpImpOrNil(cls, sel, inst, 
                            NO/*initialize*/, YES/*cache*/, NO/*resolver*/)) 
        {
            _class_resolveInstanceMethod(cls, sel, inst);
        }
    }
}

这个会判断是不是元类如果不是那么就去执行_class_resolveInstanceMethod方法，如果是元类就去执行_class_resolveClassMethod方法，会有一个lookUpImpOrNil方法的判断。

IMP lookUpImpOrNil(Class cls, SEL sel, id inst, bool initialize, bool cache, bool resolver) {
    IMP imp = lookUpImpOrForward(cls, sel, inst, initialize, cache, resolver);
    if (imp == _objc_msgForward_impcache) return nil;
    else return imp;
}

static void _class_resolveInstanceMethod(Class cls, SEL sel, id inst) {
    if (! lookUpImpOrNil(cls->ISA(), SEL_resolveInstanceMethod, cls,NO/*initialize*/, YES/*cache*/, NO/*resolver*/)) {
        // 没有找到 resolveInstanceMethod: 方法，直接返回。
        return;
    }

    BOOL (*msg)(Class, SEL, SEL) = (__typeof__(msg))objc_msgSend;
    bool resolved = msg(cls, SEL_resolveInstanceMethod, sel);

    // 缓存结果，以防止下次在调用 resolveInstanceMethod: 方法影响性能。
    IMP imp = lookUpImpOrNil(cls, sel, inst,
                             NO/*initialize*/, YES/*cache*/, NO/*resolver*/);
}

_class_resolveInstanceMethod方法执行之后就会进入goto retry就会重新进入查找方法流程，但是这样不会永远停不下来了的查找嘛？答案是不会，因为triedResolver = YES标记位已经变成了YES，不会再进入没有找到方法再去查找一下的逻辑。
